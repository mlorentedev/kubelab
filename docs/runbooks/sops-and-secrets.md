# SOPS and Secrets Management

## Overview

KubeLab uses [SOPS](https://github.com/getsops/sops) with [age](https://github.com/FiloSottile/age) encryption for secrets management. All secrets are stored encrypted in `infra/config/secrets/*.enc.yaml` and decrypted at runtime by the toolkit.

**Key principle**: secrets never exist in plaintext in git. Only encrypted files are committed.

## Architecture

```
.sops.yaml                          # Encryption rules (which keys encrypt which files)
~/.config/sops/age/keys.txt         # Private key (NEVER in git, NEVER shared unprotected)

infra/config/secrets/
├── dev.enc.yaml                    # Dev secrets (encrypted)
├── staging.enc.yaml                # Staging secrets (encrypted)
├── prod.enc.yaml                   # Production secrets (encrypted)
└── dev.oidc-jwks.pem               # OIDC signing key (generated by toolkit)
```

**How it works**:

1. `age-keygen` generates a keypair (public + private)
2. Public key goes in `.sops.yaml` (safe to commit)
3. Private key goes in `~/.config/sops/age/keys.txt` (never committed)
4. SOPS encrypts with the public key, decrypts with the private key
5. Toolkit calls `sops -d` to decrypt at runtime during `config-generate`

## Prerequisites

- `sops` installed (`brew install sops` / `apt install sops`)
- `age` installed (`brew install age` / `apt install age`)
- Private key at `~/.config/sops/age/keys.txt`

## Procedures

### 1. Initial setup (new machine)

```bash
# Install tools
# macOS
brew install sops age

# Ubuntu/Debian
sudo apt install age
# sops: download from https://github.com/getsops/sops/releases

# Generate age keypair
age-keygen -o ~/.config/sops/age/keys.txt

# Output shows public key like:
# Public key: age1xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

Then add the public key to `.sops.yaml` in the repo:

```yaml
creation_rules:
  - path_regex: infra/config/secrets/.*\.enc\.yaml$
    key_groups:
      - age:
          - "age1your_public_key_here"
```

### 2. Setup from existing project (onboarding)

If the project already has encrypted secrets and you need access:

```bash
# 1. Get the private key from the existing developer or password manager (Vaultwarden)
# 2. Place it at the correct location
mkdir -p ~/.config/sops/age/
cp /path/to/received/keys.txt ~/.config/sops/age/keys.txt
chmod 600 ~/.config/sops/age/keys.txt

# 3. Verify you can decrypt
sops -d infra/config/secrets/dev.enc.yaml
```

### 3. View secrets (read-only)

```bash
# Via toolkit (recommended)
toolkit secrets show                                    # All dev secrets
toolkit secrets show apps.services.security.authelia -e staging  # Specific key
toolkit secrets catalog                                 # List all registered secrets
toolkit secrets catalog -v                              # With descriptions

# Via sops directly
sops -d infra/config/secrets/dev.enc.yaml
```

### 4. Edit secrets

```bash
# Via toolkit (recommended — sets $EDITOR automatically)
toolkit secrets edit --env dev
toolkit secrets edit --env staging

# Via sops directly
sops infra/config/secrets/dev.enc.yaml
```

### 5. Add secrets for a new service

```bash
# 1. Generate a secret value
python3 -c "import secrets; print(secrets.token_urlsafe(64))"

# 2. Edit the SOPS file
sops infra/config/secrets/dev.enc.yaml

# 3. Add the new key in the YAML structure:
#    apps:
#      services:
#        category:
#          new_service:
#            api_key: "generated_value_here"
#            db_password: "another_generated_value"

# 4. Save and close (auto re-encrypts)

# 5. Reference in values/common.yaml or values/dev.yaml
# 6. Regenerate configs
make config-generate
```

### 6. Generate all credentials (toolkit)

The toolkit has a unified secrets pipeline. See [secrets-reference](secrets-reference.md) for the full catalog.

```bash
# Step 1: Generate machine secrets (tokens, hex keys, RSA)
toolkit secrets init --env dev

# Step 2: Generate passwords + derived hashes (interactive)
toolkit credentials generate --env dev --auto-update

# Step 3: Generate OIDC JWKS RSA key
toolkit secrets jwks --env dev

# Step 4: Hash all OIDC client secrets
toolkit secrets hash --env dev

# Step 5: Verify
toolkit secrets audit --env dev
```

### 6b. Generate staging/prod secrets

Each environment needs its own independent secrets:

```bash
# Same pipeline, different env
toolkit secrets init --env staging
toolkit credentials generate --env staging --auto-update
toolkit secrets jwks --env staging
toolkit secrets hash --env staging
toolkit secrets audit --env staging

# Apply to K8s cluster
toolkit secrets apply --env staging
```

**Important**: staging secrets must be distinct from dev and prod. Never copy dev secrets to staging — generate fresh ones.

### 7. Add a collaborator (new age key)

```bash
# 1. Collaborator generates their keypair and sends you their PUBLIC key
# 2. Add it to .sops.yaml:
#    - age:
#        - "age1existing_key..."
#        - "age1new_collaborator_key..."

# 3. Re-encrypt all secrets with both keys
sops updatekeys infra/config/secrets/dev.enc.yaml
sops updatekeys infra/config/secrets/staging.enc.yaml
sops updatekeys infra/config/secrets/prod.enc.yaml

# 4. Commit .sops.yaml and the re-encrypted files
```

### 8. Rotate secrets

See [secrets-reference](secrets-reference.md) for the standard rotation procedure.

```bash
# Full rotation for an environment
toolkit secrets init --env prod
toolkit credentials generate --env prod --auto-update
toolkit secrets jwks --env prod
toolkit secrets hash --env prod
toolkit secrets audit --env prod

# Propagate
toolkit secrets apply --env prod        # K8s secrets
toolkit config generate --env prod      # Rendered configs (if applicable)
toolkit infra k8s deploy --env prod     # Redeploy pods
```

## Recovery

### Lost private key (keys.txt)

If you lose `~/.config/sops/age/keys.txt`:

1. **You cannot decrypt existing secrets** — this is by design
2. Generate a new keypair: `age-keygen -o ~/.config/sops/age/keys.txt`
3. Update `.sops.yaml` with the new public key
4. **Regenerate ALL secrets** from scratch: `make credentials-generate`
5. Re-encrypt: `sops -e -i infra/config/secrets/dev.enc.yaml`
6. Repeat for staging and prod

**Prevention**: store `keys.txt` in your Vaultwarden instance or a secure backup location.

## Credential Model

KubeLab uses three authentication mechanisms. In dev they share the same password for convenience; in staging/prod each must be unique.

### Authentication layers

| Mechanism | What it does | Protects | Stored as |
|-----------|-------------|----------|-----------|
| **Authelia SSO** | Identity provider. Traefik forwards unauthenticated requests to Authelia login. | grafana, portainer, traefik dashboard, n8n, and any service behind `authelia` middleware | `users_admin_password_hash` (argon2id) in `users_database.yml` |
| **Basic Auth** (htpasswd) | HTTP Basic Auth via Traefik middleware. Fallback for services without native auth. | Endpoints not covered by Authelia or requiring a second factor | `basic_auth.credentials` (bcrypt htpasswd) in SOPS |
| **Per-service admin passwords** | Internal admin accounts for each service's own UI/API. | grafana admin, minio root, n8n, gitea admin | Per-service keys in SOPS (e.g., `apps.services.observability.grafana.admin_password`) |

### Secrets file structure (SOPS)

```yaml
basic_auth:
  user: admin
  password: <plaintext>           # For HTTP Basic Auth
  credentials: <htpasswd bcrypt>  # Pre-computed for Traefik

apps:
  services:
    security:
      authelia:
        users_admin_password_hash: <argon2id>   # What Authelia stores
        oidc_hmac_secret: <token>
        session_secret: <token>
        storage_encryption_key: <token>
        oidc_client_secret: <token>              # For OIDC clients (Grafana, MinIO)
        oidc_client_secret_hash: <argon2id>
        oidc_client_secret_grafana: <token>
        oidc_client_secret_grafana_hash: <argon2id>
      crowdsec:
        bouncer_api_key: <token>
    data:
      minio:
        root_user: admin
        root_password: <plaintext>
    observability:
      grafana:
        admin_user: admin
        admin_password: <plaintext>
```

### Querying secrets

```bash
toolkit credentials show                                    # All secrets (dev)
toolkit credentials show -e staging                         # All secrets (staging)
toolkit credentials show basic_auth                         # Basic Auth block
toolkit credentials show apps.services.security.authelia    # Authelia block
toolkit credentials show apps.services.data.minio           # MinIO block
```

### Dev vs staging/prod

| | Dev | Staging / Prod |
|---|---|---|
| Password reuse | All services share same password (convenience) | Each service has unique credentials |
| Generation | `toolkit credentials generate` | `toolkit credentials generate -e staging` |
| Login to Authelia | `admin` / shared password | `admin` / unique password |
| Login to Grafana | `admin` / shared password | `admin` / unique password (+ Authelia SSO via OIDC) |

### Adding credentials for a new service

1. Edit SOPS: `sops infra/config/secrets/dev.enc.yaml`
2. Add keys under `apps.services.<category>.<service_name>`
3. Reference in `infra/config/values/common.yaml` templates
4. Regenerate configs: `toolkit config generate`
5. Update this section if the new service uses a different auth mechanism

## Verification

```bash
# Can decrypt
sops -d infra/config/secrets/dev.enc.yaml | head -5

# SOPS config is valid
cat .sops.yaml

# Toolkit can read secrets
make config-generate && make validate
```

## K8s Secrets Pipeline (ADR-014)

> Ref: [adr-014-secrets-management-strategy](../adr/adr-014-secrets-management-strategy.md)

K8s secrets are generated by the toolkit from SOPS and applied directly (NOT stored in Git):

```bash
# Generate K8s secrets from SOPS and apply to cluster
toolkit infra k8s apply-secrets --env staging
toolkit infra k8s apply-secrets --env prod
```

**Key design decisions (ADR-014):**
- K8s Secret manifests are toolkit-generated, never committed to Git
- `secrets.yaml` is NOT in `kustomization.yaml` — applied separately by toolkit
- Future: migrate to Sealed Secrets after ArgoCD (Stream E: SEAL-001..004)

**Secret definitions** are in `toolkit/features/k8s_secrets.py`:

| Secret name | Namespace | Keys | Source in SOPS |
|-------------|-----------|------|----------------|
| `authelia-secrets` | kubelab | 6 keys (session, storage, OIDC, etc.) | `apps.services.security.authelia.*` |
| `grafana-admin` | kubelab | user, password | `apps.services.observability.grafana.*` |
| `grafana-oidc` | kubelab | client_id, client_secret | `apps.services.security.authelia.oidc_client_secret_grafana` |
| `crowdsec-bouncer` | kubelab | api_key | `apps.services.security.crowdsec.bouncer_api_key` |
| `api-secrets` | kubelab | (empty placeholder) | — |
