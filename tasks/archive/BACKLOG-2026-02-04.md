### TICKET-008: Persistence Layer

**Priority**: P1
**Type**: Infrastructure
**Reference**: Content integrated from former `docs/todo/PERSISTENCE_STRATEGY.md`

#### Architecture

Hybrid Persistence Model:

1. **Structured Data**: PostgreSQL 16 with multiple schemas
2. **Unstructured Data**: MinIO (S3-compatible object storage)
3. **Caching/Queues**: Redis (ephemeral)

```text
┌─────────────────────────────────────────────────────────────┐
│                     DATA LAYER                               │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐         │
│  │ PostgreSQL  │  │   MinIO     │  │   Redis     │         │
│  │   :5432     │  │   :9000     │  │   :6379     │         │
│  │             │  │             │  │             │         │
│  │ Schemas:    │  │ Buckets:    │  │ Uses:       │         │
│  │ - auth      │  │ - uploads   │  │ - Cache     │         │
│  │ - app       │  │ - reports   │  │ - Celery    │         │
│  │ - analytics │  │ - assets    │  │ - Sessions  │         │
│  │ - workers   │  │             │  │             │         │
│  └─────────────┘  └─────────────┘  └─────────────┘         │
│         │                │                │                 │
│         └────────────────┴────────────────┘                 │
│                     Backend Network                         │
└─────────────────────────────────────────────────────────────┘
```

### TICKET-020: Repository Migration Strategy

**Priority**: P2
**Type**: Architecture
**Estimate**: 15-20 hours initial + 2-3 hours/month maintenance
**Owner**: Manuel Lorente

#### Objective

Transform current monorepo into a professional, modular ecosystem:

- Maximum reusability via immutable artifacts (Docker GHCR, Releases, Go/NPM packages)
- Automated GitOps (Actions for build/publish/deploy)
- Granular security and access (GitHub Org)
- Recruiter-attractive (Platform Engineer showcase)
- Freelancer-friendly (limited access) and SaaS-ready (independent repos)

### TICKET-022: Risk Mitigation & Access Control

**Priority**: P2
**Type**: Planning

#### Identified Risks

| Risk | Mitigation | Priority |
|------|------------|----------|
| Staging downtime during migration | Test all changes in feature branch first | High |
| Secrets leak to wrong repo | Strict org roles, repo-level secrets only | High |
| Initial complexity overwhelming | Step-by-step, test each phase before next | Medium |
| CI/CD breaks during migration | Keep old workflows until new ones validated | Medium |
| Freelancers access sensitive infra | No access to platform-engine | High |

| Role | platform-engine | sensor-tool | portal | internal-libs |
|------|-----------------|-------------|--------|---------------|
| **Owner (Manuel)** | Admin | Admin | Admin | Admin |
| **Freelancer (SaaS dev)** | NO ACCESS | Write | NO ACCESS | Read |
| **Freelancer (content)** | NO ACCESS | NO ACCESS | Write | NO ACCESS |
| **GitHub Actions** | Read (pull) | Read (toolkit) | Read (toolkit) | Read |
| **Public** | NO ACCESS | NO ACCESS | Read | NO ACCESS |

| Secret | Scope | Used By |
|--------|-------|---------|

## Best Practices Applied

### 1. Compose File Pattern (compose.{base,dev,staging,prod}.yml)

**Why**: Separates shared config from environment-specific overrides.

```bash
# Usage
docker compose -f compose.base.yml -f compose.dev.yml up -d

# compose.base.yml: shared volumes, networks, healthchecks
# compose.dev.yml: dev ports, hot reload, debug flags
# compose.prod.yml: resource limits, replicas, logging
```

### 2. Directory Structure (infra/stacks/)

**Why**: Clear separation of concerns.

```
infra/stacks/
├── apps/      → Custom applications (source in apps/)
├── edge/      → Network edge (Traefik, Nginx, Cloudflared)
└── services/  → Third-party services by category
    ├── ai/
    ├── automation/
    ├── core/
    ├── data/
    ├── misc/
    ├── observability/
    └── security/
```

### 3. Service Categories

| Category | Purpose | Services |
|----------|---------|----------|
| **core** | Essential platform | gitea, portainer |
| **observability** | Monitoring/logging | grafana, loki, uptime |
| **security** | Auth/protection | authelia, crowdsec |
| **data** | Storage/docs | minio, docmost |
| **automation** | CI/workflows | github-runner, kestra |
| **ai** | ML/AI | ollama, webui |
| **misc** | Productivity | calcom, immich |

### 4. Toolkit Pattern

```
Toolkit (fat)     → Complex logic in Python
Makefile (thin)   → Bootstrap only (91 lines)
```

### 5. Environment Strategy

```
dev      → Local development (hot reload, debug)
staging  → KubeLab homelab (mirrors prod, limited resources)
prod     → Hetzner VPS (public, full resources)
```
